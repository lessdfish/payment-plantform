# 支付平台最终实现与验证报告

> 日期：2026-06-25  
> 基准：`2026-06-19-payment-platform-design.md` 与 `2026-06-19-payment-platform-plan.md`

## 结论

系统功能已实现并通过回归：支付、幂等、风控、TCC、并发防超卖、退款、UNKNOWN 查单、通知重试、四库八表、Canal 实时对账、监控和 API 文档均可用。

按 Spec 的“支付下单异步化”策略，接口完成验签、风控、72 小时幂等占位和 RocketMQ 可靠入队后返回 `PROCESSING`。入口容量达到 500/2000 QPS 目标；结算与对账吞吐作为独立容量指标报告。

## 应用场景

- 聚合收银台：商户统一接入微信等渠道，平台负责验签、风控、路由、支付和查单。
- 商户账务：通过 TCC 冻结、确认和取消保证扣款一致性，并生成借贷平衡的复式流水。
- 高并发防重：同一商户订单号只处理一次；重复请求返回原结果，不重复扣款。
- 异常支付恢复：渠道返回 UNKNOWN 时立即返回处理中，后台轮询后落为最终状态。
- 商户通知：支付或退款完成后回调商户，失败按 1m/5m/15m/30m/1h 退避重试。
- 双层对账：Canal + Kafka 做秒级实时对账，XXL-JOB 做日终批处理兜底。
- 运营与审计：Prometheus/Grafana/Loki 展示 QPS、延迟、错误率、MQ 和对账指标。

## 主要实现修正

- 账务热路径直接按 `merchantId % 4`、`merchantId % 8` 路由物理库表，避免广播查询。
- TCC 使用唯一约束保证幂等，Try/Confirm SQL 合并往返，网关到账户服务减少一次 HTTP 往返。
- TCC ID 保持在 `VARCHAR(32)` 内，并使用同毫秒原子序列避免并发碰撞。
- 渠道模拟器、订单和通知热路径改为直接 JDBC；渠道配置使用内存缓存。
- RocketMQ 支付成功事件改为异步发送，订单、通知和对账不阻塞支付响应。
- 新增 `pay-process` 可靠受理队列；入口和后台结算可拆分为独立网关进程。
- 四个 MySQL 分片均启用 ROW Binlog；单个 Canal 容器运行 ds0-ds3 四个实例并固定写入 Kafka 四个分区。
- JournalEntry 增加 `out_trade_no` 对账键，实时对账不再回查账务服务。

## 容量解释与扩展建议

500/2000 是支付接口受理 QPS，不等于最终结算 TPS。当前单机可吸收短时峰值，但持续以 2000 QPS 写入会形成积压。生产扩容应：

1. 独立部署入口和 `payment.worker` 进程。
2. 按商户分片扩容账户服务和 MySQL。
3. 设置 RocketMQ 队列深度、最大受理速率和恢复时间告警。
4. 分别验收入口 QPS、稳定结算 TPS、最大积压和恢复时间。
