## 三、10 万 ~ 30 万 QPS 的架构体系

微信/QQ/抖音/高德能做到秒级响应 + 海量并发，是因为每一层都在极致优化。

### 3.1 完整请求链路

```
用户请求
  ↓ DNS/GSLB（智能调度到最近的机房）
  ↓ CDN/边缘节点（静态资源直接返回，不穿透到源站）
  ↓ LVS/DPVS（四层负载均衡，内核态转发，单机百万连接）
  ↓ Nginx/Envoy（七层负载均衡，SSL卸载、限流、灰度）
  ↓ API Gateway（鉴权、签名、限流、路由、协议转换）
  ↓ 业务服务（无状态 + 水平扩展，几十到几千个实例）
  ↓ 多级缓存（本地Caffeine → Redis Cluster → DB）
  ↓ 消息队列（削峰填谷，核心链路同步 + 非核心异步）
  ↓ 数据库（分库分表 + 读写分离 + 异构索引）
```

### 3.2 各层具体手段

**负载均衡层**

```
- DPVS（爱奇艺开源）：基于 DPDK，单机 300 万+ 并发连接，L4 转发
- Nginx/OpenResty：SSL 卸载、WAF、动态 upstream
- 一致性哈希：同一用户路由到同一后端，提高缓存命中率
```

**网关层**

```
- 验签：预计算 + 缓存公钥，单机 5 万+ QPS
- 限流：令牌桶（本地内存计数，不每次查 Redis）
- 降级：熔断后直接返回兜底数据（如"系统繁忙"文案）
- 灰度：根据 uid hash 分流到不同版本
```

**业务服务层**

```
- 无状态设计：Session 外移到 Redis，服务可任意伸缩
- 异步化：核心链路同步返回，非核心（发通知、写日志、更新缓存）全部异步
- 线程模型：Netty/虚拟线程（JDK 21），线程数 = 核心数而非请求数
- 连接池：HTTP 连接池（PoolingHttpClient）、DB 连接池（HikariCP）、Redis 连接池
- 序列化：Protobuf 比 JSON 快 5-10 倍，字节数少 60%
```

**多级缓存**

```
L1 - CDN 边缘：静态资源（图片/CSS/JS），离用户最近的节点，命中率 > 95%
L2 - 本地缓存 Caffeine：渠道配置、费率，1 分钟刷新，命中率 > 99%
L3 - Redis Cluster：热点数据，10 万 QPS/节点，水平扩展
L4 - 数据库：只有 L1-L3 全 miss 才落到 DB，占比 < 1%
```

**数据库层**

```
- 分库分表：merchant_id 取模 → 4 库 × 8 表
- 读写分离：主库写入，从库查询（查订单、查流水）
- 异构索引：MySQL 存原始数据，Elasticsearch 建搜索索引
- 冷热分离：热数据在内存/SSD，冷数据归档到 HBase/对象存储
- NoSQL 专项：时序数据用 InfluxDB/TDengine
```

**消息队列异步削峰**

```
- 支付成功 → 同步返回"处理中" → MQ → 异步结算 → 商户回调
- 网关的 RT 只有 50ms（只做验签+入队），结算的 500ms 不阻塞用户
- 峰值 30 万 QPS 过来，MQ 堆积 100 万条，消费者慢慢消化
```

**内核与网络调优**

```
- TCP 参数：tcp_tw_reuse、tcp_max_tw_buckets、somaxconn
- 文件描述符：ulimit -n 655350
- 零拷贝：sendfile() 传输静态文件，绕过用户态
- DPDK/XDP：绕开内核协议栈直接操作网卡，单机千万 PPS
```

### 3.3 10 万 vs 30 万 QPS 的差异

| 维度 | 10 万 QPS | 30 万 QPS |
|------|----------|----------|
| 业务机数量 | 50-100 台 (8C16G) | 200-400 台 |
| Redis 集群 | 3 主 3 从足够 | 10+ 主，数据分片 |
| MySQL | 8 库分片基本够 | 16-32 库，读写分离更激进 |
| MQ 分区 | 8-16 分区 | 32-64 分区 |
| 网络 | 万兆网卡 | 多网卡 bonding |
| 是否需要自研 | 开源组件调优即可 | 可能自研网关/缓存/序列化 |

### 3.4 当前项目与十万级 QPS 的差距

```
当前项目：
  Gateway → REST 同步调用 4 个服务 → 通道返回 → TCC → 响应
  一套请求 ≈ 70ms，QPS 上限 ≈ 2000（Sentinel 限流）

十万级架构改造：
  核心思路：把同步链路拆成"同步受理 + 异步结算"

  改造后：
  Gateway 收到支付请求
    → 验签 + 幂等 + 风控 + 入 MQ（纯内存操作，~5ms）
    → 立即返回"PROCESSING"
    → 消费者异步处理：渠道扣款 → TCC → 写流水 → 商户回调
    → 商户通过查询接口获知最终结果

  改造后 QPS：
  Gateway 纯内存操作 5ms → 200 线程可支撑 40,000 QPS（单机）
  10 台 Gateway = 400,000 QPS
```

> 当前项目 `application.yml` 中已预留 `payment.async.enabled` 和 `payment.worker.enabled` 配置项，说明异步化改造方向已在设计范围内。

## 四、单商户并发 vs 多商户分散

### 单商户并发有意义的地方

```java
// 50 个线程同时扣 A 商户的 1000 元余额
// 余额只有 1000，只能有一笔成功，其余 49 笔必须被乐观锁挡住
UPDATE account SET balance = balance - 1000, version = version + 1
WHERE merchant_id = 1 AND version = 3
// 49 个线程拿到的 version 已经变了 → WHERE 命中 0 行 → 返回余额不足
```

**这验证的是：防超卖 + 乐观锁正确性。** 这是支付系统的最核心正确性保证。

### 单商户并发没意义的地方

压测**吞吐量**时，应该用**多商户分散**。因为：

1. 真实场景 1000 个商户各自发起支付，不会挤在一个商户上
2. 单商户数据全落在一个分片（ds${merchant_id % 4}），人为制造热点
3. 多商户分散到不同分片，才能测出真实吞吐

### 正确做法

```
场景设计原则：
  ├─ 一致性验证：单商户 + 多线程 + 同一余额，验证乐观锁/幂等/无超卖
  └─ 吞吐验证：多商户 + 多线程 + 各自余额，验证分片均衡下的真实 QPS
```

**单商户并发不是无意义的**，它测的是"最坏情况下的正确性"。只是不能拿这个数据当作系统的吞吐上限。

## 五、结论

1. 互联网业务方向 30+ 个，支付项目落在最有竞争力的 Top 3
2. 各领域技术栈不同，Java 统治支付/金融/电商/风控
3. 10 万 QPS 的核心是"同步受理 + 异步结算 + 多级缓存 + 分片"，当前项目已埋了异步化的种子
4. 单商户并发测正确性，多商户分散测吞吐，两个维度缺一不可
